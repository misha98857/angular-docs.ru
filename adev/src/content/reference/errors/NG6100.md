# Анти-паттерн: присваивание module.id свойству NgModule.id

Использование `module.id` в качестве `id` для NgModule является распространенным анти-паттерном и, скорее всего, не
несет никакой пользы в вашем коде.

NgModules могут быть объявлены с использованием `id`:

```typescript
@NgModule({
  id: 'my_module'
})
export class MyModule {}
```

Объявление `id` делает NgModule доступным для поиска с помощью операции `getNgModuleById()`. Эта функциональность
используется редко, в основном в очень специфических сценариях сборки при ленивой загрузке NgModules без получения
прямых ссылок на них. В большинстве случаев в коде Angular следует использовать динамический `import()` из ES (
`import('./path/to/module')`), так как он предоставляет прямую ссылку на загружаемый NgModule без необходимости
использования побочного эффекта глобальной регистрации.

Если вы не используете `getNgModuleById`, вам не нужно указывать `id` для ваших NgModules. Указание `id` имеет
существенный недостаток: оно делает NgModule неподверженным Tree Shaking, что может повлиять на размер вашего бандла.

В частности, паттерн указания `id: module.id` возникает из-за неправильного понимания `@NgModule.id`. В ранних версиях
Angular иногда требовалось включать свойство `moduleId: module.id` в метаданные `@Component`.

Использование `module.id` для `@NgModule.id`, вероятно, является следствием путаницы между `@Component.moduleId` и
`@NgModule.id`. `module.id` обычно бесполезен для операций `getNgModuleById()`, так как `id` должен быть заранее
известной строкой, а `module.id` обычно непрозрачен для потребителей.

## Отладка ошибки

Вы можете удалить объявление `id: module.id` из ваших NgModules. Компилятор игнорирует это объявление и вместо этого
выдает данное предупреждение.
